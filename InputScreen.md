# 📋 要件定義書：入力画面（確定版・猫イラスト追加）

## 1. 概要（目的とゴール）

* **目的：** ユーザーが解析したい麻雀の牌譜URL（雀魂・天鳳）を迷わずに入力でき、バックエンドへ正しくデータを送信すること。
* **ゴール：** 正しいURLが入力された場合はスムーズにローディング・結果画面へ遷移し、不適切な入力に対しては分かりやすいエラーを表示してユーザーに再入力を促せる状態を作る。

## 2. ユーザー体験（UX）とユースケース

* **想定される流れ：**
1. ユーザーがアプリの「入力画面（トップ画面）」を開く。
2. URL入力フォームに、自分の雀魂または天鳳の牌譜URLを手動（コピペ）で入力する。
3. 「解析を開始する」ボタンをクリックする。
4. 入力内容のチェック（バリデーション）が走り、問題がなければバックエンドへAPI送信を行い、ローディング状態へ移行する。



## 3. 機能要件（フロントエンド担当）

### 3.1 表示項目（画面に載せるもの）

* [ ] 画面タイトル（「麻雀 AI牌譜添削ツール（仮）」）
* [ ] URL入力フォーム（テキストボックス、プレースホルダーとして「ここにURLをペースト」と表示）
* [ ] 解析開始ボタン
* [ ] 対応サイトの案内注記（フォームのすぐ上に「※雀魂・天鳳の牌譜URLに対応しています」とテキスト表示）
* [ ] **猫のイラスト（※MVP対象外：画面の空きスペースに配置する癒やし要素。リリース優先度低）**

### 3.2 入力仕様とバリデーション

* **対象項目：** 雀魂（じゃんたま）または天鳳（てんほう）の牌譜URL
* **必須 / 任意：** 必須入力
* **文字数・形式制限（バリデーション規則）：**
* フロントエンド側で簡易チェックを実施。文字列が `https://` で始まり、かつドメイン部分に `mahjongsoul` または `tenhou.net` のいずれかが含まれていることを条件とする。



### 3.3 データ表示・表現形式

* **表示の粒度：** 入力画面のため、基本は1つのフォームとボタンのみのシンプルな構成。
* **データ項目：** 送信ボタン押下時に、入力された文字列をそのままバックエンドAPIに引き渡す。

## 4. 非機能要件・エラーハンドリング

### 4.1 エラーケースの定義

* [ ] **未入力エラー：** フォームが空の状態でボタンが押された場合、フォームの下に赤字で「URLを入力してください」と表示し、API送信を中断する。
* [ ] **不正な形式エラー：** 雀魂・天鳳以外のURLや文字列が入力された場合、フォームの下に赤字で「雀魂または天鳳の有効な牌譜URLを入力してください」と表示し、API送信を中断する。
* [ ] **通信エラー（API失敗）：** サーバーがダウンしている等で通信自体が失敗した場合、フォームの下に赤字で「サーバーとの通信に失敗しました。時間をおいて再度お試しください」と表示する。

### 4.2 状態（ステート）の定義

* [ ] **初期状態（Empty State）：** フォームは空欄。ボタンは活性化（クリック可能）状態。
* [ ] **ローディング状態：** ボタン押下後、API通信が始まったらボタン内のテキストを「解析中...」に変更し、連打防止のためボタンを非活性化（クリック不可）にする。

## 5. バックエンド・LLM側への要望（I/F要件）

* **画面表示に必要なデータ（フロントエンドから送るもの）：**
* [ ] ユーザーが入力したURL（文字列：`{ "url": "xxxx" }` のようなJSON形式を想定）


* **バックエンドから返してほしいデータ（成否）：**
* [ ] **成功時：** ステータスコード200 ＋ 解析データ（JSON）
* [ ] **サーバー側エラー時：** 「URLの牌譜が既に削除されている」「非公開になっている」など、サーバー側でエラーを検知した場合は、エラーコード（例：400 Bad Request）と一緒に `{ "error_message": "該当の牌譜データが見つかりませんでした" }` というメッセージを返してもらい、フロントエンド側はそれをそのままフォーム下に赤字で表示する。
